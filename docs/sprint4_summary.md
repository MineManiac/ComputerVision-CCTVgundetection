# Sprint 4 Summary - Hold/No-Hold Gate Redesign

Sprint 4 implemented a two-phase CCTV weapon detection pipeline in which the first stage detects people, crops each detected person with additional context, and classifies that crop as `hold` or `no_hold` before sending approved candidates to the Sprint 3 `YOLO26n` detector. The original two-phase experiment used a hard threshold selected by validation F1 and showed that Stage 1 became the dominant bottleneck: too many person crops were rejected before the weapon detector could inspect them, causing a severe recall collapse relative to the single-stage baseline.

The redesigned Sprint 4 workflow keeps the same overall architecture but changes the semantics and policy of Phase 1. User-facing outputs now describe the first stage as `hold/no_hold`, the default person-crop padding is larger to preserve more context around the body and the held item, and the gating rule is now intentionally permissive. Instead of choosing the Stage 1 cutoff only by best F1, the training script now saves both a reporting threshold (`best_f1_threshold`) and an operational threshold (`stage1_gate_threshold`). The operational threshold is selected with a recall-floor policy so that more candidates reach Phase 2 and the pipeline is less likely to collapse because of an overly strict first-stage gate.

As a result, Sprint 4 should now be reported as three comparable scenarios rather than a single two-phase result: the single-stage YOLO26n baseline, the older two-phase run using the F1-selected gate, and the redesigned two-phase run using the permissive `hold/no_hold` gate. The expected tradeoff is explicit: Stage 1 may allow more false positives to pass temporarily, but this is preferable to discarding true positives before Phase 2 can make the final weapon decision. The final recommendation still depends on rerunning the full evaluation with the redesigned gate and checking whether the additional candidate flow improves recall enough to justify the staged architecture.
